# 04-0 LLM 三塔向量召回与语义个性化

## 1. 本章导读

### 1.1 召回层在整体架构中的位置

前面的系统已经搭建了 AgentLoop 决策层：主 AgentLoop 通过
`Think -> Act -> Observe -> Reflect` 循环调用工具，遇到复杂子任务时通过
`task_tool` fork 同质子 AgentLoop。

当主 AgentLoop 调用：

```text
item_search(query="旅行收纳袋")
```

`item_search` 需要从约 1.5 亿件商品中找到最相关的几十件。其内部核心不是简单的
关键词匹配，而是向量召回：将查询和商品编码到同一个向量空间，再进行 ANN
（Approximate Nearest Neighbor，近似最近邻）检索。

## 2. 为什么需要向量召回

### 2.1 传统关键词匹配的局限

传统电商搜索通常以 BM25 或 TF-IDF 为核心，根据关键词是否出现在商品文本中进行
匹配。这种方式存在三个主要问题：

| 问题 | 示例 | 根因 |
| --- | --- | --- |
| 同义不匹配 | 用户搜索“收纳袋”，商品标题使用“整理包” | 词面不同就可能无法匹配 |
| 意图不理解 | 用户搜索“出差带的东西” | 无法从词面直接理解场景意图 |
| 个性化缺失 | 两个用户搜索同一个词时结果完全一致 | 没有建模用户长期偏好 |

### 2.2 向量召回的解决方式

向量召回把 Query 和 Item 映射到同一个向量空间，用向量相似度替代单纯的词面匹配：

- “收纳袋”和“整理包”在向量空间中的距离较近，可以实现同义召回。
- “出差带的东西”可以被编码为包含“旅行”“便携”“商务”等概念的向量。
- Query 向量可以融合用户长期偏好，使相同查询对不同用户返回不同结果。

## 3. 三塔模型：User / Query / Item

### 3.1 为什么使用三个塔

传统双塔模型通常只有 Query 塔和 Item 塔。电商搜索中，用户长期偏好与当前查询的
即时意图是两种不同信号，因此拆分为三个独立编码器：

- **User 塔**：编码用户长期偏好，包括历史行为、购买频率和品类倾向。
- **Query 塔**：编码当前查询表达的即时意图。
- **Item 塔**：编码商品文本、属性和历史交互特征。

三个塔分别编码，最终通过统一向量空间中的相似度完成匹配。

### 3.2 各塔输入与目标

| 塔 | 输入信息 | 编码目标 |
| --- | --- | --- |
| User | 点击、收藏、购买等用户历史行为序列，以及用户画像 | 生成代表用户长期偏好的向量 |
| Query | 当前搜索词，以及该查询下历史成交商品关键词等 RAG 增强信息 | 生成代表本次搜索意图的向量 |
| Item | 商品标题、类目、属性和历史成交 Query | 生成代表商品特征的向量 |

### 3.3 请求向量与 ANN 检索

检索时先融合 User 向量和 Query 向量，形成请求向量，再与 Item 向量库做 ANN 检索：

```text
request_vector = f(user_vector, query_vector)
results = ANN_search(request_vector, item_vector_index, top_k=100)
```

## 4. 解耦式双通道：语义与个性化

### 4.1 为什么需要解耦

如果语义相关性和个性化转化共用一个训练目标，容易出现指标“跷跷板”：

- 加强语义相关性时，个性化指标可能下降。
- 加强个性化时，召回商品可能偏离当前 Query。

两类目标的梯度方向经常冲突。例如，语义相关性要求“搜索旅行收纳袋时返回收纳袋”，
个性化则可能因用户长期偏爱高端商品而倾向高价款。将两类信号直接耦合，容易让模型
在当前意图与长期偏好之间失衡。

### 4.2 双通道设计

系统将两个任务拆成独立通道，分别训练，在请求侧融合后完成一次 ANN 检索：

| 通道 | 训练目标 | 输入 | 输出 |
| --- | --- | --- | --- |
| 语义通道 | Query-Item 文本相关性 | Query 与 Item 文本 | 语义相关性向量 |
| 个性化通道 | User-Query-Item 转化预估 | User、Query 与 Item 特征 | 个性化向量 |

融合公式为：

```text
final_vector = alpha * semantic_vector + beta * personalization_vector
results = ANN_search(final_vector, item_vector_index, top_k=100)
```

`alpha` 和 `beta` 控制当前语义与长期偏好的相对权重。两路向量必须具有相同维度、
归一化约定和可比较的尺度，才能进行线性融合并进入同一个 Item ANN 索引。

## 5. 与 AgentLoop 的关系

完整调用链如下：

```text
用户输入“旅行收纳袋”
  -> 主 AgentLoop Think：需要搜索商品
  -> Act：item_search(query="旅行收纳袋")
    -> item_search 内部：
       1. Query 塔编码当前查询
       2. User 塔编码当前用户历史
       3. 融合为请求向量
       4. 在 Item 向量库执行 ANN Top-100 检索
       5. 返回结构化商品列表
  -> Observe：获得 100 件候选商品
  -> Think：调用 ItemPicker 按偏好与约束筛选
```

对主 AgentLoop 来说，三塔和 ANN 是 `item_search` 的内部实现细节。主循环只依赖稳定的
工具输入输出契约，不需要感知底层使用几个塔或哪一种 ANN 引擎。

召回质量决定后续排序的上限。如果 Top-100 候选本身不相关，ItemPicker 和 Reranker
都无法从缺失的候选中恢复正确商品。

## 6. 多任务训练

三塔模型采用多任务联合训练，不只依赖单一 Loss：

| 任务 | 正样本 | 负样本 | 作用 |
| --- | --- | --- | --- |
| 成交任务 | 用户实际购买的商品 | 曝光但未成交的商品 | 学习转化信号 |
| 点击任务 | 用户点击的商品 | 进入精排但未点击的商品 | 学习兴趣信号 |
| 曝光任务 | 已曝光商品 | 未曝光商品 | 学习基础覆盖 |
| 精排一致性 | 精排高分商品 | 精排低分商品 | 对齐召回与后续排序 |

三塔召回模型与 AgentLoop 主模型属于两条独立训练链路，各自优化自己的业务指标，
通过 `item_search` 工具契约衔接。

## 7. 本章小结

1. `item_search` 的内部实现是向量召回，而不是单纯关键词匹配。
2. User 塔建模长期偏好，Query 塔建模即时意图，Item 塔建模商品特征。
3. 语义与个性化双通道分别训练，通过线性融合共同形成请求向量。
4. 请求向量在 Item 向量库中执行 ANN，并返回 Top-100 候选。
5. 三塔召回对主 AgentLoop 保持封装，与 AgentLoop 主模型独立训练和演进。
